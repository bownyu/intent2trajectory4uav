# intent2trajectory 当前代码实现整理

本文整理当前仓库已经落地的代码实现，重点说明五种意图的轨迹生成方式，以及从配置读取到 CSV 输出、再到 GUI 可视化的完整代码流程。

## 1. 当前实现范围

当前仓库实现的是“运动学阶段”的数据生成器，不包含 JSBSim/PX4 仿真闭环控制。本阶段的核心目标是：

- 根据配置批量生成五类意图轨迹。
- 为每条轨迹输出标准化 CSV。
- 输出 `metadata.csv` 记录生成参数、随机种子和状态。
- 对样本做动力学与行为约束校验，不合格则重采样。
- 提供一个 GUI 工具，用于从目录中选择 CSV 并按 CSV 时间播放 3D 轨迹。

核心代码文件：

- `src/intent2trajectory/generator.py`
- `src/intent2trajectory/visualization.py`
- `scripts/trajectory_player_gui.py`
- `configs/dataset_config.json`

## 2. 生成器总体结构

### 2.1 对外接口

`src/intent2trajectory/generator.py` 暴露三个核心接口：

- `generate_dataset(config_path)`：读取配置并批量生成整个数据集。
- `generate_sample(intent, seed, profile)`：生成单条指定意图的样本。
- `validate_sample(sample, profile)`：校验单条样本是否满足物理约束和意图特征。

辅助模块 `src/intent2trajectory/visualization.py` 提供：

- `list_csv_files(input_dir)`：递归列出目录下所有 CSV。
- `load_trajectory_csv(csv_path)`：读取单个 CSV，解析可视化所需时序与位置列。
- `compute_intervals_ms(times, speed)`：将 CSV 时间列转换为 GUI 播放间隔。

### 2.2 批量生成的完整代码流程

`generate_dataset(config_path)` 的执行顺序如下：

1. 调用 `load_config(config_path)` 读取 JSON/YAML 配置。
2. 从配置中取出：
   - `output_root`
   - `class_quota`
   - `max_resample_attempts`
   - `seed`
3. 按固定顺序 `INTENT_ORDER` 创建五类输出目录：
   - `0_hover`
   - `1_straight_penetration`
   - `2_non_straight_penetration`
   - `3_loiter`
   - `4_retreat`
4. 对每一类意图、每一个样本编号，执行重采样循环：
   - 通过 `_stable_seed(base_seed, intent, sample_index, attempt)` 派生稳定随机种子。
   - 调用 `generate_sample(...)` 生成单条样本。
   - 调用 `validate_sample(...)` 做约束校验。
   - 如果通过则接受该样本。
   - 如果失败则记录失败原因，继续下一次 attempt。
5. 如果某个样本在最大尝试次数内始终失败：
   - 不生成轨迹 CSV。
   - 在 `metadata.csv` 中写入 `status=failed` 和 `failure_reason`。
6. 如果样本通过校验：
   - 从元数据中提取初始距离和平均速度。
   - 生成文件名：`[intent]_[序号]_D[初始距离]_V[平均速度].csv`
   - 调用 `_write_sample_csv(...)` 写入轨迹 CSV。
   - 将该样本的关键参数写入 `metadata.csv`。
7. 最终返回：
   - `generated`
   - `failed`
   - `output_root`

### 2.3 单样本生成的完整代码流程

`generate_sample(intent, seed, profile)` 的流程如下：

1. 检查 `intent` 是否在 `INTENT_ORDER` 中。
2. 用 `random.Random(seed)` 构造本样本独立随机数生成器。
3. 从配置中读取：
   - `dt`
   - `max_time`
   - 对应意图的 `intent_profiles[intent]`
4. 根据意图类型分发到具体生成函数：
   - `hover` -> `_gen_hover(...)`
   - `straight_penetration` -> `_gen_straight(...)`
   - `non_straight_penetration` -> `_gen_non_straight(...)`
   - `loiter` -> `_gen_loiter(...)`
   - `retreat` -> `_gen_retreat(...)`
5. 具体生成函数返回：
   - `positions`：三维位置序列
   - `yaw_policy`：本样本采用的航向策略
   - `extra_noise`：本样本的噪声/几何参数摘要
6. 生成时间序列：
   - `times = [i * dt for i in range(len(positions))]`
7. 用 `_derivative(positions, dt)` 对位置序列做离散差分，得到速度序列 `velocities`。
8. 用 `_derive_yaw(...)` 根据意图和速度计算航向角序列 `yaws`。
9. 将每个时刻整理为一行 CSV 记录，字段包括：
   - `time_relative`
   - `pos_x`, `pos_y`, `pos_z`
   - `yaw`
   - `vel_x`, `vel_y`, `vel_z`
   - `ref_pos_*`, `ref_yaw`, `ref_vel_*`
   - `act_pos_*`, `act_yaw`, `act_vel_*`
10. 当前阶段 `act_*` 一律写为 `NaN`，为后续仿真回填预留位置。
11. 构建样本级元数据：
   - `seed`
   - `start_xyz`
   - `end_xyz`
   - `base_speed`
   - `duration`
   - `yaw_policy`
   - `noise_profile`
   - `constraint_profile`
   - `initial_distance`
   - `end_distance`
12. 返回样本对象：
   - `sample_id`
   - `intent`
   - `rows`
   - `metadata`

## 3. 五种意图的轨迹生成方式

下面按代码中的实际实现说明五种意图是如何生成的。

### 3.1 悬停 `hover`

对应函数：`_gen_hover(ip, dt, max_time, rng)`

#### 生成逻辑

1. 从 `duration_range` 中随机采样时长，并受 `max_time` 上限约束。
2. 从三维空间范围中随机采样一个悬停中心点：
   - `x ~ U(space.x)`
   - `y ~ U(space.y)`
   - `z ~ U(space.z)`
3. 从 `noise_sigma` 中采样扰动强度 `sigma`。
4. 初始点设置为中心点 `center`。
5. 后续每个时刻都在上一个点基础上更新：
   - 对 `x/y/z` 三个方向分别加入高斯扰动
   - 同时对当前位置到中心点的偏差乘一个衰减因子 `0.995`
6. 结果是一个围绕中心点的低速、小幅、可回归的随机游走过程。

#### 代码特征

- `x/y` 扰动缩放为 `0.012 * sigma`
- `z` 扰动缩放为 `0.007 * sigma`
- 更新公式本质是：
  - `当前位置 = 中心点 + 残余偏差 + 小扰动`
- `yaw_policy` 固定为 `"fixed"`
- `noise_profile` 记录为 `gaussian_walk_sigma_xxx`

#### 航向生成

在 `_derive_yaw(...)` 中：

1. 先随机采样一个固定朝向 `fixed`。
2. 再叠加非常小的高斯噪声 `yaw_noise_sigma`。
3. 用 `_wrap_angle(...)` 将角度规范化到 `[-pi, pi]`。

因此，悬停样本的特征是：

- 空间上基本停在一点附近
- 速度很低
- 航向基本稳定，只带极小扰动

### 3.2 直线突防 `straight_penetration`

对应函数：`_gen_straight(ip, dt, max_time, rng)`

#### 生成逻辑

1. 调用 `_sample_polar_xyz(...)` 在极坐标空间随机采样起点：
   - 半径 `R ~ U(start_radius)`
   - 方位角 `theta ~ U(0, 2pi)`
   - 高度 `z ~ U(start_z)`
2. 构造从起点指向原点的单位向量：
   - `u = unit(-start)`
3. 从 `base_speed` 中采样基础速度 `speed`。
4. 每个时间步沿 `u` 方向前进：
   - 速度上叠加一个小的正弦扰动 `0.3 * sin(0.03 * t)`
5. 一直推进到以下任一条件触发：
   - 到达 `target_radius`
   - 超过 `max_time`

#### 代码特征

- 核心是固定方向匀速逼近原点
- 只允许速度幅值有很小波动，不改变整体方向
- `yaw_policy` 标记为 `"towards_origin"`
- `noise_profile` 标记为 `"sin_jitter"`

#### 航向生成

在 `_derive_yaw(...)` 中：

1. 直接根据起点到原点的方向计算一个常量 yaw：
   - `atan2(-y0, -x0)`
2. 将这个 yaw 复制到整个轨迹长度。

因此，直线突防样本的特征是：

- 距离原点持续减小
- 轨迹近似直线
- 航向几乎完全稳定

### 3.3 非直线突防 `non_straight_penetration`

对应函数：`_gen_non_straight(ip, dt, max_time, rng)`

#### 生成逻辑

1. 与直线突防一样，先随机采样远距离起点。
2. 构造指向原点的单位方向向量 `u`。
3. 构造与 `u` 在水平面内垂直的横向向量 `w`：
   - `w = unit((-u_y, u_x, 0))`
4. 从配置中采样：
   - 基础前向速度 `speed`
   - 横向振幅 `amp`
   - 横向周期 `period`
5. 计算横向角频率：
   - `omega = 2pi / period`
6. 为避免横向机动过强导致速度超限，对振幅做二次裁剪：
   - `amp = min(amp, max_lateral_speed / abs(omega))`
7. 每个时间步的位置由三部分叠加构成：
   - 沿 `u` 向前推进：`speed * t`
   - 沿 `w` 横向正弦摆动：`amp * sin(omega * t)`
   - 在高度方向加一个较小的正弦波动：`0.02 * amp * sin(0.7 * omega * t)`
8. 一直推进到：
   - 距离原点小于 `target_radius`
   - 或超过 `max_time`

#### 代码特征

这个实现本质上是“向目标逼近 + 横向蛇形机动”的组合模型：

- 前向分量保证总体趋近目标
- 横向分量使局部路径持续弯折
- 纵向分量幅度较小，只提供轻微高度变化

`noise_profile` 会记录为：

- `sin_amp_[振幅]_T_[周期]`

#### 航向生成

在 `_derive_yaw(...)` 中：

1. `yaw_policy` 是 `"velocity_tangent"`
2. 每个时刻用速度矢量计算航向：
   - `yaw = atan2(vy, vx)`

因此，非直线突防样本的特征是：

- 距离原点整体减小
- 局部轨迹不是直线
- 航向随运动方向持续变化

### 3.4 盘旋 `loiter`

对应函数：`_gen_loiter(ip, dt, max_time, rng)`

#### 生成逻辑

1. 先随机采样盘旋中心点 `center`：
   - 半径来自 `center_radius`
   - 高度来自 `center_z`
2. 随机采样盘旋半径 `radius`。
3. 随机采样线速度 `speed`。
4. 根据线速度和半径计算角速度：
   - `omega = speed / radius`
5. 随机决定盘旋方向：
   - 顺时针或逆时针
6. 根据 `duration_range` 和一个完整圆周周期 `period` 计算盘旋圈数：
   - `loops = randint(min_loops, max_loops)`
7. 对每个时刻，根据几何形状生成位置：
   - 若 `shape == "circle"`：
     - `x = cx + r cos(omega t + phi0)`
     - `y = cy + r sin(omega t + phi0)`
   - 若 `shape == "figure8"`：
     - 使用 8 字形式的参数方程
8. 当前默认 `z = center_z`，即高度保持不变。

#### 代码特征

默认配置中：

- `shape` 为 `"circle"`
- `yaw_policy` 为 `"tangent"`

`noise_profile` 会记录盘旋方向：

- `loiter_circle_cw`
- `loiter_circle_ccw`

#### 航向生成

在 `_derive_yaw(...)` 中：

1. `yaw_policy` 为 `"tangent"`
2. 所以直接按速度方向计算 yaw：
   - `yaw = atan2(vy, vx)`

因此，盘旋样本的特征是：

- 与目标距离整体不发生大幅净变化
- 轨迹呈圆形或 8 字形闭合结构
- 航向沿切线方向持续旋转

### 3.5 撤退 `retreat`

对应函数：`_gen_retreat(ip, dt, max_time, rng)`

#### 生成逻辑

1. 从较近距离范围随机采样起点：
   - 半径来自 `start_radius`
   - 高度来自 `start_z`
2. 构造从原点指向起点的单位向量 `u`，即“远离原点”的方向。
3. 从 `base_speed` 中采样撤退速度 `speed`。
4. 每个时间步都按以下方式推进：
   - 主方向速度：`u * speed`
   - 叠加一个很小的高斯扰动 `noise`
5. 当以下条件之一满足时停止：
   - 距离大于 `end_radius`
   - 超过 `max_time`

#### 代码特征

这个模型是带前向偏置的随机游走：

- 主体方向始终背离原点
- 小扰动只负责增加自然性，不改变总体趋势
- `noise_profile` 记录为 `"biased_walk"`

#### 航向生成

在 `_derive_yaw(...)` 中：

1. `yaw_policy` 为 `"velocity_tangent"`
2. 所以每个时刻的航向与速度方向一致。

因此，撤退样本的特征是：

- 首末距离显著增大
- 整体轨迹由近向远扩散
- 航向随运动方向变化，但总体保持“远离目标”

## 4. 轨迹后处理与状态量计算

### 4.1 速度计算

速度并不是在各意图函数里直接生成到 CSV，而是统一通过 `_derivative(points, dt)` 计算：

- 对于非最后一个点：
  - `v_i = (p_{i+1} - p_i) / dt`
- 对于最后一个点：
  - `v_last = (p_last - p_prev) / dt`

这种设计把“位置生成”和“速度导出”解耦，所有意图统一使用相同的速度计算方式。

### 4.2 航向计算

航向统一由 `_derive_yaw(...)` 生成，当前支持三类逻辑：

- 固定朝向加微扰：主要用于 `hover`
- 指向原点：主要用于 `straight_penetration`
- 切向/速度方向：用于 `non_straight_penetration`、`loiter`、`retreat`

### 4.3 元数据

单条样本会记录如下元数据：

- 随机种子
- 起点/终点坐标
- 平均速度
- 总时长
- 航向策略
- 噪声模式
- 约束配置名称
- 初始距离与末端距离

这些信息最终会进入总账 `metadata.csv`。

## 5. 样本校验流程

`validate_sample(sample, profile)` 用来决定一条样本能否进入数据集。

### 5.1 通用物理约束

所有意图都会检查：

- 空域边界：
  - `x` 是否在 `constraints.space.x` 范围内
  - `y` 是否在 `constraints.space.y` 范围内
  - `z` 是否在 `constraints.space.z` 范围内
- 最大速度：
  - 使用 CSV 中的 `vel_x/vel_y/vel_z` 计算速度模长
- 最大加速度：
  - 用相邻时刻速度差分除以 `dt`
- 最大偏航角速度：
  - 用相邻时刻 `yaw` 差值除以 `dt`

### 5.2 行为级约束

在通用约束之外，各意图还会检查自己的行为特征。

#### hover

- 最大偏离中心距离不能过大
- 平均速度不能过高

#### straight_penetration

- 末距离必须小于初距离
- 航向标准差必须很小

#### non_straight_penetration

- 末距离必须小于初距离
- 净逼近量至少达到阈值
- 航向标准差不能过小，否则会被视为过于接近直线突防

#### loiter

- 首末距离差不能过大，否则说明出现明显径向漂移

#### retreat

- 末距离必须满足：
  - `end_distance >= distance_multiplier * initial_distance`

### 5.3 重采样机制

如果校验失败：

- `generate_dataset(...)` 不会直接接受该样本
- 而是继续换一个派生随机种子重试
- 直到成功或达到 `max_resample_attempts`

因此，最终落盘的数据集是“生成 + 校验 + 重采样”共同作用的结果。

## 6. CSV 输出结构

每条样本 CSV 由 `generate_sample(...)` 逐行组织，当前字段包括：

- 时间：
  - `time_relative`
- 主状态：
  - `pos_x`, `pos_y`, `pos_z`
  - `yaw`
  - `vel_x`, `vel_y`, `vel_z`
- 参考轨迹：
  - `ref_pos_x`, `ref_pos_y`, `ref_pos_z`
  - `ref_yaw`
  - `ref_vel_x`, `ref_vel_y`, `ref_vel_z`
- 预留给仿真回填的实际轨迹：
  - `act_pos_x`, `act_pos_y`, `act_pos_z`
  - `act_yaw`
  - `act_vel_x`, `act_vel_y`, `act_vel_z`

当前阶段：

- `pos_*` 与 `ref_*` 内容相同
- `act_*` 为 `NaN`

这样做的目的是让未来引入仿真闭环时，可以直接沿用同一份 CSV 结构，而不需要重设计数据格式。

## 7. 可视化代码流程

可视化由两个部分组成：

- 数据解析模块：`src/intent2trajectory/visualization.py`
- GUI 播放器：`scripts/trajectory_player_gui.py`

### 7.1 CSV 解析流程

`load_trajectory_csv(csv_path)` 的逻辑如下：

1. 以 `utf-8-sig` 读取 CSV，兼容 BOM。
2. 用 `csv.DictReader` 解析每一行。
3. 选择时间列：
   - 优先 `time_relative`
   - 其次 `time`
4. 选择位置列，优先级是：
   - `act_pos_x/y/z`
   - `ref_pos_x/y/z`
   - `pos_x/y/z`
5. 过滤掉无效数值行。
6. 按时间排序并返回：
   - `times`
   - `xs`
   - `ys`
   - `zs`
   - 以及使用了哪一组列

这意味着，当未来 `act_*` 被仿真回填后，GUI 会自动优先播放实际轨迹；当前阶段则自动回退到 `ref_*`。

### 7.2 GUI 播放流程

`scripts/trajectory_player_gui.py` 里的 `TrajectoryPlayerApp` 工作流程如下：

1. 启动时传入 `input_dir`，并调用 `refresh_file_list()` 扫描目录下所有 CSV。
2. 用户在左侧列表中选择一个文件后，点击 `Load`。
3. `load_selected()` 调用 `load_trajectory_csv(...)` 读取数据。
4. 用 `compute_intervals_ms(times, speed)` 将 CSV 时间差转成 GUI 播放间隔。
5. 点击 `Play` 后，进入 `_schedule_next()` 循环：
   - 每次根据当前帧索引调用 `_draw_frame(idx)`
   - 画出截至当前帧的整段轨迹
   - 再用 `root.after(interval, ...)` 预约下一帧
6. `_draw_frame(idx)` 中：
   - 蓝线显示已飞过轨迹
   - 红点显示当前飞行器位置
   - 状态栏显示当前时间、帧数和播放倍率

### 7.3 时间播放机制

GUI 不是按固定帧率播放，而是按 CSV 实际时间差播放：

- 若相邻样本时差为 `0.1s`
- 且播放倍率为 `1.0x`
- 则 GUI 间隔为 `100ms`

若用户提高播放倍率：

- `speed=2.0`
- 同样的 `0.1s` 间隔会变成 `50ms`

也就是说，GUI 的播放速度与轨迹时间轴是严格对应的，而不是简单动画。

## 8. 当前实现的关键设计选择

### 8.1 先生成位置，再统一求速度和航向

这样做的好处：

- 五类意图共享同一套状态导出逻辑
- 更容易调试某个意图的几何轨迹
- 以后接入仿真时也更容易对齐 `ref` 与 `act`

### 8.2 用重采样而不是复杂修复

当前实现没有对不合格样本做复杂“修补”，而是：

- 生成
- 检查
- 不通过就丢弃重采样

这种做法简单、可控，而且容易保证最终样本分布质量。

### 8.3 用统一 CSV 结构给仿真预留接口

即使当前没有 `actual` 数据，也提前保留：

- `act_pos_*`
- `act_yaw`
- `act_vel_*`

这保证后续从“运动学生成器”切到“运动学 + 仿真回填”时，不会破坏下游读取逻辑。

## 9. 当前代码的完整主链路总结

如果从“生成一批数据并进行可视化”这个角度看，当前完整链路如下：

1. 用户调用：
   - `generate_dataset('configs/dataset_config.json')`
2. 代码读取配置、创建五类目录。
3. 对每个类别按配额循环生成样本。
4. 对每个样本：
   - 选择对应意图生成函数
   - 生成位置序列
   - 差分得到速度
   - 生成航向
   - 组织 `rows` 和 `metadata`
5. 执行 `validate_sample(...)`：
   - 先检查物理上限
   - 再检查意图特征
6. 通过则写入：
   - 样本 CSV
   - `metadata.csv`
7. 用户再调用 GUI：
   - `python scripts/trajectory_player_gui.py --input-dir dataset_workspace`
8. GUI 扫描目录内 CSV，加载一条样本。
9. GUI 自动选择可用位置列并读取时间列。
10. 按 CSV 的真实时间间隔播放三维飞行轨迹。

以上就是当前仓库已经实现的完整代码流程。
